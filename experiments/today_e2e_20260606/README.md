# 今日端到端 integration — 第一次全自动管线（2026-06-06）

第一次跑通完全自动的管线：collect（代码采集）→ synthesize（DS-V4 自动挑瞬间写 prompt）→ image（gpt-image-2 出图）。今天 8–20 点六个窗口并行。

## 结果：管线通了，但"自动挑瞬间"失败了

- **技术上**：6 窗口里 5 张出图（w4 被 gpt-image-2 output moderation 误挡，偶发，可重试）。collect→DS-V4→image 全自动串通，DS-V4 输出格式完全合规（单场景/鸭哥/彩铅/3:4/6色/无文字）。本地 DS-V4 调用正常。
- **内容上：六张几乎是同一张画**——鸭哥戴眼镜指着一块电子屏、旁边树莓派、桌上散代码纸。它把"做这块 eink 日记屏"这一件事画了六遍，没有记录今天不同时段的不同经历。**作为日记失败。**

## 根因（比技术故障重要）

1. **DS-V4 不会"挑瞬间"**：当前 system prompt 缺两样——跨窗口去重/多样化约束；以及"别挑最高频主题，要挑有情绪张力的具体细节"。它看到素材里满屏都是 eink 项目讨论，就每个窗口都挑了这个最高频主题。
2. **时序精度问题发作**：ai_sessions 无逐条时间戳 → 当天所有 session 内容涌进每个窗口。今天 session 几乎全是 eink 项目讨论，于是六窗口拿到的素材高度雷同，DS-V4 自然挑出一样的东西。微信本可提供窗口区分，但今天白天微信发言被 session 洪流淹没。
3. **对比昨天**：昨天八张各不相同且贴合真实，是因为**人**手工从素材里挑了不同的有张力的点。今天交给 DS-V4 自动挑，暴露了这一步的判断质量不足。

## 价值

端到端实验的意义正在于：把"自动挑瞬间"这个之前用手工绕过的难点，暴露成了下一个要解决的真问题。架构（采集层代码 + 判断层 AI，边界在素材文件）是对的，管线是通的，缺的是 synthesize 的"挑瞬间"判断力。

## 下一步候选

- 给 synthesize 加多样化/去重约束（如：一次性看全天、为 8 个窗口各挑一个**互不相同**的瞬间；或挑瞬间时显式排除"项目元讨论"这类元主题）。
- 解决 session 时序精度（上游导出器补消息级时间戳），让每个窗口的素材真正不同。
- 引入健康数据源，给"微信沉默"的窗口补真实状态。
